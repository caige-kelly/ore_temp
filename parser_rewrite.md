
## README: The Ore Compiler Architecture (Step 15.5)

### Core Philosophy: "The Compiler as a Database"

Ore is designed with **Mechanical Sympathy** at its core. Instead of a traditional "Tree" of pointers, Ore treats source code as a set of relational tables. This allows for near-instant incremental updates and a low-memory footprint.

### 1. The Data-Oriented AST

The AST is split into two distinct lifetimes: **Durable** (Semantic logic) and **Volatile** (Source location).

* **Durable Data (The Body):** A set of parallel arrays containing `Kinds`, `Children`, and `Values`. Because these do not store file positions, they remain bitwise identical even if code "slides" around due to whitespace edits.
* **Volatile Data (The SpanMap):** A side-table mapping `NodeId -> Span`. This is recomputed on every parse and provides the "Live" location of code for diagnostics and the LSP.

### 2. The "Big Four" Side-Tables

To keep AST nodes lean, metadata is moved into specialized side-tables. This ensures the CPU cache is only ever filled with the data needed for the current pass.

| Table | Mapping | Purpose |
| --- | --- | --- |
| **SpanMap** | `Id -> ByteRange` | Maps nodes to their current location in the source. |
| **ParentMap** | `Id -> Id` | Allows $O(1)$ upward traversal (essential for LSP and Scoping). |
| **TriviaMap** | `Id -> [Token]` | Stores comments and whitespace for lossless LSP formatting. |
| **TypeMap** | `Id -> TypeId` | Stores the results of the Type Inference pass. |

### 3. The Query System & Early Cutoff

Ore uses a demand-driven query system inspired by Salsa.

1. **Durable Hashing:** After a parse, the Durable AST is fingerprinted.
2. **Memoization:** Queries (like `query_resolve_ref`) are cached based on the `NodeId` and the AST's fingerprint.
3. **Early Cutoff:** If a reparse results in an AST with the same fingerprint as the previous one, all downstream queries are immediately marked as valid. No type-checking or codegen is re-run.

### 4. LSP & Comptime Integration

* **Lossless Fidelity:** By using the `TriviaMap`, Ore can reconstruct the original source perfectly, making "Format on Save" and "Refactor" features easy to implement.
* **Virtual Spans:** The `Span` structure uses a `FileId` that can point to either a physical file on disk or a virtual buffer generated by `comptime` logic. This allows generated code to be treated as a first-class citizen in diagnostics.

### 5. Implementation Constraints

* **No Raw Pointers:** AST nodes must never hold raw pointers to other nodes. Use `NodeId` exclusively.
* **Value-Oriented Errors:** Diagnostics are stored in a `DiagBag` per-query and use `NodeId` instead of `Span` to ensure they "slide" with the code.
* **Parallel Arrays over Unions:** Favor parallel arrays for node data to maximize cache-line utilization during semantic passes.

---

### Why this works for Ore:

By decoupling the "What" (semantics) from the "Where" (spans), we achieve the performance of **Odin**, the incremental stability of **Rust-Analyzer**, and the mechanical sympathy required for high-performance **Effect Systems** and **Comptime** execution.

### The "Step 15.5" Green-Light Summary

Before you tear into the code, here is your final "Pre-Flight" checklist to ensure the refactor goes smoothly:

#### 1. Lexer & Layout: The "Lossless" Goal

* **Trivia Support:** Ensure the lexer/layout combo can produce "Trivia" (whitespace/comments). Even if you don't use it today, having the hook in the `Parser` to push trivia into a side-table is what makes your LSP "Gold Tier" later.
* **Virtual Tokens:** Your layout engine is already Koka-inspired; just make sure the `indent`/`dedent` tokens it generates are treated as first-class citizens by the new parser.

#### 2. Parser: The "Parallel Array" Shift

* **ID-First:** Replace `struct Expr*` returns with `ExprId`.
* **The Big Three:** Implement the `Kinds[]`, `Children[]`, and `Values[]` arrays.
* **Side-Table Push:** Instead of `e->span = t.span`, use `p->spans[id] = t.span`.

#### 3. Bridge to Sema

* **Accessor Functions:** Write a small set of helpers like `expr_kind(s, id)` and `expr_child(s, id, index)`. This prevents you from having to rewrite every line of Sema. You just change the *implementation* of how Sema looks at a node.
* **Span Lookup:** Update your diagnostic calls to look up the span from the `SpanMap` using the `NodeId`.

### One Final Tip: The "Two-Phase" Parse

While you're refactoring, consider splitting your `parse_source` into two internal phases:

1. **Structural Parse:** Just build the tree (the parallel arrays).
2. **Parent Mapping:** A quick, linear post-pass that fills the `ParentMap`.
Doing this in a post-pass is often faster and cleaner than trying to track "current_parent" through every recursive-descent function in the parser.
